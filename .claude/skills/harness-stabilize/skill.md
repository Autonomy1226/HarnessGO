# ⛔ 硬边界：你只设计补稳机制。核心产出：开发与验证之间的硬咬合。

你在 Harness 8 阶段流水线中的位置：**第 6 步 / 共 8 步**。

你的**最重要任务**：确保流水线中「开发→评审→测试」三个环节不是松散的「建议」，而是**不通过就卡住的硬门禁**。

## 你绝对不能做的事

- ❌ 不要运行任何 Agent 或测试
- ❌ 不要修改 Agent 角色定义
- ❌ 不要写业务代码

## 执行步骤

### 第 1 步：定义开发→评审的硬咬合

生成开发阶段的**提交前检查清单**，写入 `docs/harness/stabilization.md`：

```markdown
## 阶段间硬咬合规则

### 开发实现 → 代码评审（前进条件）
开发 Agent 申请进入评审前，PM Agent 必须验证以下**全部**通过：
- [ ] `scripts/verify.sh` 第 2 步（编译）退出码为 0
- [ ] `scripts/verify.sh` 第 3 步（测试）退出码为 0
- [ ] 所有新增/修改文件已在 `docs/harness/dev-map.json` 中注册
- [ ] 开发日志 `docs/dev/dev-log.md` 已记录

**任何一项不通过 → 不得进入代码评审。开发 Agent 修复后重新申请。**
```

### 第 2 步：定义评审→测试的硬咬合

```markdown
### 代码评审 → 测试验证（前进条件）
评审 Agent 完成后，PM Agent 验证：
- [ ] 审查报告 `docs/reviews/review-report.md` 存在
- [ ] 审查报告中的结论为「通过」(无阻塞项)
- [ ] 如有严重问题 → 回退到开发阶段，不得进入测试

### 测试验证 → 交付（前进条件）
测试 Agent 完成后，PM Agent 验证：
- [ ] `scripts/verify.sh` 全量通过（5/5，退出码为 0）
- [ ] SPEC 验收标准逐条通过，记录在测试报告中
- [ ] 测试报告 `docs/tests/test-report.md` 存在
- [ ] verify.sh 中无占位符、无 SKIP（全部为 PASS 或 FAIL）
```

### 第 3 步：定义回退规则

```markdown
## 回退规则

| 触发条件 | 回退目标 | 回退后动作 |
|----------|----------|-----------|
| 闸门驳回 | 方案设计 | 方案设计 Agent 修正 + 重新过闸 |
| 评审发现阻塞项 | 开发实现 | 开发 Agent 修复 + 重新提交评审 |
| verify.sh 失败 | 开发实现 | 开发 Agent 定位失败步骤 + 修复 + 重新验证 |
| SPEC 验收标准不通过 | 开发实现 | 开发 Agent 补全 + 重新验证 |
```

### 第 4 步：Agent 输出校验

对 8 个角色逐一定义 Schema/完整性/一致性校验规则。

### 第 5 步：容错策略

- 关键路径（开发、测试）：阻塞 + 重试 2 次 → 人工介入
- 非关键路径（评审）：失败记录 → 降级但标记
- 幂等性：state.json + 产出物路径去重

### 第 6 步：生成配置

写入 `docs/harness/stabilization.md`。
更新 `docs/harness/state.json`：`stabilize` 标记 `completed`，`current_phase` 设为 `"verify"`。

## ⛔ 完成标准

- [ ] 回退规则表完整
- [ ] 每个咬合规则都有明确的验证命令（不依赖人工判断）
