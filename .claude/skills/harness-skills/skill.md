# ⛔ 硬边界：你只识别和生成 Skill 骨架

你在 Harness 8 阶段流水线中的位置：**第 4 步 / 共 8 步**。
上一个阶段是 `harness-rules`，下一个是 `harness-agents`。

## 为什么要把编译、测试、验证单独做成 Skill

工程里有一类事情，非常适合做成 Skill：

| 特征 | 为什么 |
|------|--------|
| 执行步骤固定 | 不需要 AI 每次重新推理 |
| 每次都要做 | 是底线，不是可选 |
| 做错一次代价大 | 漏了编译 → 上线炸了；漏了测试 → 回归没发现 |
| 不值得让 AI 重新思考 | AI 应该把精力花在业务逻辑上，不是花在「怎么跑 pytest」上 |

编译、单元测试、事后验证——就是这一类事情。

## Rule 与 Skill 的分工

```
Rule：你必须编译、测试、事后验证。三步全过才算完成。
Skill：编译怎么编、测试怎么跑、验证怎么验。一步一步写清楚。
```

**Rule 变轻，Skill 变稳。**

具体好处：

1. **Rule 变轻了**——以前 Rule 既讲原则又讲流程，越写越长。现在 Rule 只保留红线，复杂的执行步骤都在 Skill 里。

2. **AI 的执行稳定性变高了**——以前 AI 需要临场理解「编译应该怎么编、测试命令是什么」，每次都可能写出不同的命令。现在不是理解——是调用一套现成的方法，每次跑出的结果都是一模一样的。

3. **维护成本更低了**——后来你修改编译流程，只需要改编译 Skill 本身，不需要去全项目搜索每条 Rule 里是不是都写了旧命令。

## 你必须生成的 4 个标准 Skill

### Skill 1: 编译验证 (build-verify)

```
触发条件：代码变更后
执行步骤：
  1. 检测项目构建命令（package.json build / Makefile / cargo build / go build）
  2. 执行构建
  3. 收集错误输出并分类（语法/类型/依赖/其他）
  4. 判定：零错误 = PASS，任何错误 = FAIL
输出：[PASS] 编译通过 或 [FAIL] + 错误分类
错误处理：构建命令不存在 → 询问配置；超时 → 建议拆分
```

### Skill 2: 测试验证 (test-verify)

```
触发条件：编译通过后
执行步骤：
  1. 检测项目测试命令
  2. 运行全量测试，收集失败用例
  3. 计算覆盖率，对比 Rules 中定义的最低阈值
  4. 判定：全量通过 + 覆盖率达标 = PASS
输出：[PASS] 测试通过 (覆盖率: X%) 或 [FAIL] + 失败用例列表
错误处理：测试环境未就绪 → 提示配置；失败用例超 N → 建议分批修复
```

### Skill 3: 事后验证 (post-verify)

```
触发条件：测试通过后
执行步骤：
  1. 读取 SPEC 中的成功标准（AC 验收条件）
  2. 逐条对照实现检查
  3. 检查 Non-goals 是否被意外实现
  4. 判定：AC 全部通过 + Non-goals 未越界 = PASS
输出：[PASS] 事后验证通过 或 [FAIL] + AC 未达标项
```

### Skill 4: 格式检查 (lint-check)

```
触发条件：代码变更后（可与编译并行）
执行步骤：
  1. 检测项目格式化命令
  2. 运行 Lint + Formatter
  3. 判定：零告警 = PASS
输出：[PASS] 格式检查通过 或 [FAIL] + 告警列表
```

## 执行步骤

### 第 1 步：读取 SPEC 和 CLAUDE.md Rules

### 第 2 步：确认标准 4 Skill + 项目特定 Skill

向用户确认：「除了编译、测试、事后验证、格式检查这 4 个，有部署脚本、数据迁移、API 契约校验等其他固定流程吗？」

### 第 3 步：为每个 Skill 生成 SKILL.md

生成 `.claude/skills/{skill-name}/SKILL.md`。每个文件必须包含：
- 触发条件（什么时候调用）
- 输入契约（需要什么参数）
- 执行步骤（固定操作序列）
- 输出格式（PASS/FAIL + 详细信息）
- 错误处理策略（失败了怎么办，谁负责修复）

### 第 4 步：更新 CLAUDE.md

在 Skills 节添加引用。每条引用对应 CLAUDE.md Rules 节中的一句话规则。

### 第 5 步：更新状态

`skills` → `completed`，`current_phase` → `"agents"`。

## ⛔ STOP — Skill 骨架生成完就停。不要讨论 Agent 编排。
