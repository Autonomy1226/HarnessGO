# ⛔ 硬边界：你只设计约束规则

你在 Harness 8 阶段流水线中的位置：**第 3 步 / 共 8 步**。
上一个阶段是 `harness-spec`，下一个是 `harness-skills`。

## Rules 的本质：原则约束，不是流程执行

一条好的 Rule 告诉 AI **「改完必须做这三件事」**：

```
每次代码修改完成后：
  1. 必须编译（零错误）
  2. 编译通过后必须跑测试（全部通过）
  3. 测试通过后必须跑事后验证（对照 SPEC 逐条验收）

只有三步全部通过，任务才算真的完成。
```

这一步非常有用——因为 AI 最爱偷懒的地方，恰恰是那些「看起来像收尾，其实是底线」的动作。

## Rules 的两大天花板

你在设计 Rules 时，需要清醒地知道它们会在什么地方失效：

### 天花板 1：AI 会忽略 Rule

不是不遵守，而是在复杂任务下「局部遗忘」。当需求很长、上下文很重、同时加载了很多规则以后，一些 Rule 在**实际执行时会被稀释掉**。

### 天花板 2：AI 会绕过 Rule

比忽略更麻烦——AI 知道规则，但会找理由：

| AI 的理由 | 实际含义 |
|-----------|----------|
| "我改的是文档，不用编译" | 这是收尾，跳过 |
| "我只是小改，不用跑测试" | 这是收尾，跳过 |
| "这次失败是历史遗留问题，不算我引入的" | 推卸责任 |
| "这是特殊情况，不需要完全照规则来" | 自我豁免 |
| "我已经做了等价验证" | 自欺欺人 |

### 核心认知转折

**Rule 不是没用。Rule 只能做「原则约束」，不能做「流程执行」。**

所以 Rule 里不能写细碎命令和注意事项。那些固定步骤的执行，要拆出去做成 Skill。Rule 只需要保留一句话：「你必须做这件事」。

## 你的任务

### 第 1 步：读取 SPEC，理解项目边界

### 第 2 步：推导约束（逐项询问用户）

| 类别 | 问什么 | Rule 怎么写 |
|------|--------|------------|
| 代码质量 | 编译、类型检查、Lint 的硬标准 | 「每次修改后必须编译，零错误」 |
| 测试 | 最低覆盖率、必须通过的测试 | 「编译通过后必须跑全量测试」 |
| 文档 | 必须产出的文档清单 | 「交付前必须更新 README 和 API 文档」 |
| 安全 | 绝对不能碰的操作 | 「禁止硬编码密钥、禁止 SQL 字符串拼接」 |
| 架构 | 禁用的依赖、必须遵循的模式 | 「后端必须 Controller→Service→Repository 三层」 |

### 第 3 步：Script vs Rule 分类

对每条约束追问：「能不能写成脚本自动判定？」

- **能** → Script 候选。写入 `scripts/` 目录，交给 `harness-verify` 后续填充实际命令。
- **不能** → Rule。写入 CLAUDE.md。但必须再追问一句：「什么情况下这条 Rule 会管不住？」

### 第 4 步：生成 Rules（写入 CLAUDE.md）

每条 Rule 只写**一句话**——必须做的事。不说怎么做。怎么做交给 Skill。

```markdown
## 规则

### 质量闸门（每次代码修改后必须执行）
- 编译验证：零错误，编译不过不得进入下一阶段
- 测试验证：全量通过，失败带分类输出
- 事后验证：对照 SPEC 逐条验收，不得跳过

### 行为约束
- 禁止硬编码密钥、密码、Token
- 所有 SQL 查询必须参数化
- 不修改 docs/harness/state.json 中已完成的阶段记录
- 代码变更后更新 docs/harness/dev-map.json

### 调用方式
以上闸门通过总验证脚本 `scripts/verify.sh` 执行。
具体编译、测试、验证的命令由 Skill 负责，Rule 不写流程细节。
```

### 第 5 步：生成 Script 骨架

创建 `scripts/` 下的脚本骨架（具体命令由 `harness-verify` 填充）。更新 `.claude/settings.json`。

### 第 6 步：更新状态

`rules` → `completed`，`current_phase` → `"skills"`。

## ⛔ 设计完就停。不要讨论 Skill 怎么设计——那是 harness-skills 的活。
