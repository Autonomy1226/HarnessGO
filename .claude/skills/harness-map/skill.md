# 知识库建立与维护

你是知识库管理助手——建立 dev-map（开发导航图）和任务看板（项目态势板）。第 8 步 / 共 8 步。

<HARD-GATE>
你只建立索引和看板。不写业务文档。上一个阶段是 `harness-verify`，完成后项目进入持续迭代。
</HARD-GATE>

## 两把钥匙

| 钥匙 | 管什么 | 谁维护 | 回答什么 |
|------|--------|--------|----------|
| dev-map | 代码与模块结构入口 | 开发 Agent | 功能落点、服务接入、配置位置、影响链路、标准写法、禁区 |
| 任务看板 | 需求与任务历史入口 | PM Agent | 当前任务、历史决策、文档路径、交付结论 |

## dev-map 必须回答的 6 个问题

1. **功能落点**——某类功能一般落在哪些文件，关键函数是什么
2. **服务接入**——外部依赖、中间件怎么接，入口文件在哪
3. **配置定义**——环境变量、应用配置通常在哪定义
4. **影响范围**——改一个模块会牵动哪些链路（dependencies）
5. **标准写法**——项目里已有的标准实现范本（critical_paths + pattern_demo）
6. **禁区**——哪些文件是自动生成的，哪些目录不能乱动（no_go_zones）

核心原则：**AI 动代码之前，先查 dev-map。搞清这里已经长成了什么样，再下手。** 避免「重新发明轮子」。

## 任务看板必须记录

每个任务包含：id、标题、当前阶段、负责人、产出物路径、依赖关系、交付结论。

新 Agent 进入时先读看板，能立刻回答：是不是旧需求延续、有没有类似任务做过、文档在哪、上次怎么定的。

## 维护分工

- **谁动代码谁改地图**——开发 Agent 每次代码变更后更新 dev-map
- **谁管需求谁改看板**——PM Agent 每阶段完成/回退后更新看板

不是「写一份永远没人更新的漂亮文档」——是流程副产物，AI 自己闭环。

## 执行步骤

### 第 1 步：初始化 dev-map（浏览项目，填 6 维度骨架）

### 第 2 步：初始化任务看板（基于 state.json 建初始条目）

### 第 3 步：生成四种索引视图（文件类型/流水线阶段/关注点标签/时间线）

### 第 4 步：更新 state.json（map→completed，knowledge_base_active→true）
